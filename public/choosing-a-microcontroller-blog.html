Choosing Your First Microcontroller (or SBC) for Robotics

Raspberry Pi, ESP32, Orange Pi, Arduino, STM32, Jetson — they get lumped together as "the board," but they solve completely different problems. Here's how to actually pick.

KEY CONCEPTS: micro-ROS ROS 2 Nodes/Topics/Services

1. The Real Question Isn't "Which Board" — It's "Which Category"

Every one of these boards gets recommended for "robotics" in some YouTube video, which makes them sound interchangeable. They aren't. There are three fundamentally different categories, and the first decision is which one you actually need:

  • Bare-metal microcontroller (MCU) — no operating system. Your code is the whole program, running directly on the chip. Arduino Uno/Mega, ESP32, and STM32 (Blue Pill/Nucleo) live here. You get deterministic timing and dirt-cheap hardware, at the cost of doing everything yourself — no filesystem, no threads-for-free, no apt install.
  • Linux single-board computer (SBC) — a full computer that happens to be small. Raspberry Pi 4/5 and Orange Pi 5 live here. You get a real OS, ROS 2 running natively, and a package manager — at the cost of a non-real-time kernel and GPIO timing that jitters under load.
  • AI-accelerated SBC — a Linux SBC with a GPU/NPU built specifically for running neural networks fast. NVIDIA Jetson lives here. Same OS-level tradeoffs as a Pi, plus a hardware accelerator that changes what's actually feasible for on-device vision.

Most real robots use two of these together — a Pi-class "brain" for planning/vision, talking over serial or micro-ROS to an MCU handling the motors — which is exactly the split I used across my own MechaBot and BLE localisation projects below.

2. The Boards, One by One

Arduino Uno / Mega — the on-ramp

ATmega328P (Uno) or ATmega2560 (Mega) @ 16 MHz, 2 KB/8 KB SRAM, no built-in wireless. The Uno has 14 digital I/O (6 PWM); the Mega has 54 (15 PWM) for projects juggling many servos or sensors on one board. Officially €29.30 (Uno) and €52.80 (Mega) at the Arduino Store — clones run far cheaper.

Classic Uno/Mega are not on the officially supported micro-ROS Arduino board list (that list targets newer boards — Due, Zero, UNO R4 WiFi, Portenta, Giga R1). They're too small on RAM/flash for the XRCE-DDS stack. If you want them talking to ROS 2 at all, it's a hand-rolled serial bridge, not a first-class node.

Best for: learning GPIO, PWM, and sensor reading with the simplest toolchain and the largest tutorial base on the internet. Not a real ROS 2 citizen.

ESP32 — the wireless workhorse

Dual-core Xtensa LX6 @ 240 MHz, 520 KB SRAM, 34 GPIOs, and — the reason it dominates hobbyist robotics — WiFi + Bluetooth/BLE built in, per the official Espressif datasheet. The LEDC peripheral gives 16 independent PWM generators, comfortably enough for real motor/servo control. Dev boards run about $10.

Unlike the Arduino Uno, the ESP32 (and S2/S3/C3 variants) is officially supported by micro-ROS — see the official hardware support page. That means it can be a genuine ROS 2 node, bridged through a micro-ROS Agent running on a Pi or laptop, not just a serial peripheral.

Best for: wireless sensor nodes, telemetry, and low-level actuator control that needs WiFi/BLE — this is exactly what I used it for in the BLE indoor localisation project below.

STM32 (Blue Pill / Nucleo) — real-time, done properly

ARM Cortex-M3 @ up to 72 MHz, 64 KB flash, 20 KB SRAM, 37 GPIOs across 4 ports, 3 general-purpose timers plus 1 advanced timer for tight PWM control. No built-in wireless. The "Blue Pill" (STM32F103C8T6) is an unofficial ~$2 clone board; the official ST Nucleo-F103RB has the same chip plus a built-in ST-Link debugger and Arduino-compatible headers, for roughly $10–$34 depending on distributor.

micro-ROS support exists but is strongest on the F4/F7 Nucleo family rather than this classic F103 — check the supported hardware list before committing to a specific STM32 line for a ROS 2 project.

Best for: deterministic, low-jitter control — precise PWM, encoder reading, anything where Arduino's 8-bit performance and lack of real hardware timers becomes the bottleneck. A natural step up once you've outgrown Arduino but don't need Linux.

Raspberry Pi 4 / 5 — the brain

Pi 4: quad-core Cortex-A72 @ 1.8 GHz, 1–8 GB RAM. Pi 5: quad-core Cortex-A76 @ 2.4 GHz, 1–16 GB RAM, plus PCIe 2.0 and an onboard RTC. Both have 802.11ac WiFi + Bluetooth 5.0 built in and the same 40-pin header (official Pi 5 product page). Pi Foundation launch pricing for the 5 was $50/$60/$80/$120 across the 2/4/8/16 GB variants — treat current street prices as volatile and check at purchase time.

This is the one board here that runs full, native ROS 2 — a 64-bit Ubuntu install on a Pi is Tier 1 supported per REP 2000, the official ROS platform-support policy. REP 2000's tiers are defined per-OS (Ubuntu, Debian, RHEL, etc.) and don't mention Raspberry Pi OS at all — which in practice is exactly why the standard advice is to run 64-bit Ubuntu on the Pi rather than Raspberry Pi OS if you want first-class ROS 2 support. The catch: only 2 true hardware-PWM channels on the header, and Linux isn't a real-time OS, so anything needing tight, jitter-free motor timing belongs on an MCU instead, talking to the Pi over serial or micro-ROS.

Best for: the robot's "brain" — Nav2, SLAM, OpenCV/vision pipelines, high-level planning. This is what ran the navigation stack on MechaBot and TurtleBot3.

Orange Pi 5 — the budget alternative, with caveats

Rockchip RK3588S, 8-core (4×Cortex-A76 + 4×Cortex-A55) up to 2.4 GHz, with a built-in 6 TOPS NPU — genuinely more raw compute and on-device AI acceleration than a Pi 5 for less money (the 8 GB variant lists around $199 through distributors). It has a smaller 26-pin GPIO header versus the Pi's 40, and built-in WiFi + Bluetooth 5.0.

Worth knowing before you buy: Orange Pi's official documentation and first-party support are noticeably thinner than Raspberry Pi's — expect to lean on community forums more, and to double-check exact pricing/specs directly with a distributor rather than trusting a cached spec sheet. It isn't officially listed in ROS 2's REP 2000 platform tiers, though since it runs standard arm64 Ubuntu, full ROS 2 is technically feasible the same way it is on any arm64 Ubuntu machine.

Best for: a Pi-class Linux brain on a tighter budget, or where the onboard NPU is genuinely useful — accepted in exchange for a smaller community and less polished docs.

NVIDIA Jetson Orin Nano — when you need real inference

Skip the original Jetson Nano. NVIDIA's own product lifecycle page lists the Jetson Nano Developer Kit under boards that have reached end-of-life; it's not a sensible "buy new" choice in 2026. The current entry point is the Orin Nano.

The Jetson Orin Nano Super Developer Kit: 6-core Cortex-A78AE up to 1.7 GHz, a 1024-core Ampere GPU with 32 Tensor Cores, 8 GB LPDDR5, up to 67 TOPS INT8 — for $249 official. Same 40-pin header form factor as a Pi, but no WiFi/Bluetooth built into the module itself (the dev kit carrier board has an M.2 slot for an add-in card). It runs full native ROS 2 on Ubuntu like the Pi, plus NVIDIA's Isaac ROS packages that are hardware-accelerated for its GPU.

Best for: real-time deep-learning inference — object detection, segmentation, multi-camera perception — anywhere a Pi's CPU-only vision pipeline is the bottleneck. Overkill if your project doesn't actually run a neural net on-device.

4. A Simple Decision Framework

  1. Just blinking LEDs, reading a sensor, or driving a motor with no wireless need? Arduino Uno. Cheapest on-ramp, biggest tutorial base.
  2. Need WiFi/BLE telemetry from a sensor node, or motor control with wireless, cheaply? ESP32.
  3. Need genuinely deterministic, low-jitter real-time control and can handle a steeper toolchain? STM32 — start with an official Nucleo board, not a bare clone.
  4. Need to run Linux, Nav2/SLAM/OpenCV, or act as the "brain" over serial/micro-ROS? Raspberry Pi 4/5 — or Orange Pi 5 if budget-constrained and comfortable with thinner documentation.
  5. Need real-time deep-learning inference on-device? Jetson Orin Nano. Skip the original Nano — it's EOL.

5. What I'd Actually Recommend

If you're brand new: start with an Arduino Uno for a week of GPIO/PWM basics, then move to an ESP32 the moment you want wireless — it's a small jump in complexity for a large jump in capability, and it's what I used for the BLE indoor localisation system. The moment your project needs a real navigation stack, computer vision, or anything ROS 2-native, add a Raspberry Pi as the brain rather than trying to force a Linux-shaped workload onto a microcontroller — that split is exactly what runs MechaBot: a Pi-class host running Nav2/SLAM, talking over a custom ros2_control hardware interface to embedded motor firmware. Only reach for a Jetson once you've actually confirmed your project needs on-device neural network inference faster than a Pi's CPU can deliver — it's the most expensive board here and the least useful if you don't need its GPU.
