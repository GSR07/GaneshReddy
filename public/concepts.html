Fundamentals

ROS 2: Nodes, Topics & Services

ROS 2 programs are organized as independent processes called nodes that communicate over three core primitives: topics (async pub/sub streams, e.g. sensor data), services (sync request/reply calls, e.g. "spawn a turtle"), and actions (long-running goals with feedback and cancellation, e.g. navigate to a pose). A node can be a publisher, subscriber, service client, and server all at once — the DDS middleware handles discovery and delivery so nodes never need to know about each other directly.

SLAM (Simultaneous Localization and Mapping)

SLAM solves the chicken-and-egg problem of building a map of an unknown environment while simultaneously tracking the robot's own position within it, using only noisy sensor data (LiDAR, wheel odometry, IMU). Scan-matching algorithms align successive laser scans against the growing map and correct for drift with loop closures whenever the robot revisits a previously mapped area.

AMCL (Adaptive Monte Carlo Localization)

A particle-filter localization algorithm that estimates a robot's pose on a pre-built map by maintaining a cloud of weighted pose hypotheses ("particles"). Each particle moves per the motion model and is re-weighted by how well a LiDAR scan matches the map from that hypothetical pose; resampling concentrates particles around the most likely pose over time, and the particle count adapts with localization confidence — hence "adaptive."

Appears in: MechaBot TurtleBot3 SLAM & Nav
Navigation

Nav2 (ROS 2 Navigation Stack)

Nav2 is ROS 2's production navigation framework: a global planner computes a path across the costmap (A*/Dijkstra-based), a local controller (e.g. Regulated Pure Pursuit or DWB) tracks that path while avoiding dynamic obstacles in real time, and a behavior-tree executive coordinates planning, recovery behaviors, and goal handling as a state machine.

Appears in: MechaBot TurtleBot3 SLAM & Nav
Manipulation

TF2 Coordinate Transforms

TF2 is ROS 2's library for tracking the position and orientation of every frame (camera, gripper, base, world) relative to every other frame over time, as a tree of transforms. Instead of hand-computing coordinate conversions, any node can ask "where was the camera relative to the robot base 200 ms ago?" and TF2 interpolates the answer from buffered transform history.

Appears in: Franka Panda Pick & Place
Manipulation

MoveIt2 Motion Planning

MoveIt2 is ROS 2's motion-planning framework for robotic arms: given a target end-effector pose, it plans a collision-free joint trajectory through a sampling-based planner (e.g. OMPL's RRTConnect), checks it against the robot's kinematic model and the world's collision geometry, and hands the result to a controller for execution.

Appears in: Franka Panda Pick & Place 4-DOF Robotic Arm

ros2_control Hardware Abstraction

ros2_control decouples high-level ROS 2 controllers (e.g. diff-drive, joint trajectory) from the actual hardware. A hardware interface exposes command/state interfaces — like "velocity command" and "position feedback" — over a fixed contract, so the identical controller stack can run unmodified against a Gazebo simulation or real motor firmware.

Appears in: MechaBot
Embedded

micro-ROS (ROS 2 on Microcontrollers)

Bare-metal microcontrollers like the ESP32 or an STM32 don't have the RAM or OS to run full ROS 2. micro-ROS solves this with a lightweight client library (rclc) over XRCE-DDS — a compact wire protocol that a "micro-ROS Agent" running on a full ROS 2 machine (a Raspberry Pi, laptop, etc.) bridges back into the normal ROS 2 graph, so a $10 microcontroller node can publish/subscribe like any other node.

Appears in: Choosing Your First Microcontroller for Robotics

Behavior Trees for Robot Autonomy

A behavior tree is a hierarchical way to structure decision-making: leaf nodes represent actions/conditions ("compute path", "is battery low?"), and composite nodes (Sequence, Fallback) route control flow between them, ticking the tree at a fixed rate. Nav2 uses one to coordinate planning, execution, and recovery — more composable than a monolithic state machine or nested if/else logic.

Appears in: MechaBot
Control

Proportional (P) Control

A proportional controller outputs a correction proportional to the current error: command = Kp × error. For heading control the error must be wrapped into (−π, π] (via atan2) so the robot always turns the short way around — the same P-on-distance, P-on-heading structure scales from a simulated turtlesim chase up to real differential-drive navigation.

Appears in: Turtlesim Catch-Them-All
Vision

HSV Color Segmentation

Converting an image from RGB to HSV (hue, saturation, value) separates color identity (hue) from lighting/brightness, which makes it far more robust to isolate an object by color under variable lighting than thresholding raw RGB. A hue/saturation range defines a binary mask, which contour detection then turns into object centroids and bounding boxes.

Appears in: Franka Panda Pick & Place Automated Grading & Sorting
Manipulation

Denavit–Hartenberg Kinematics (FK/IK)

D-H parameters give a standard 4-parameter-per-joint recipe (link length, twist, offset, angle) for building the chain of transforms from a robot arm's base to its end effector. Forward kinematics multiplies these transforms to get end-effector pose from joint angles; inverse kinematics solves the reverse problem, either in closed form (geometric, elbow-up/down branches) or numerically (Jacobian-based iteration).

Appears in: 4-DOF Robotic Arm
Navigation

Path Planning (Dijkstra, A*, RRT)

Path planners search a robot's configuration space for a collision-free route from start to goal. Grid-based graph search (Dijkstra, A*) is optimal and complete on discretized maps but scales poorly in high dimensions; sampling-based planners (RRT, PRM) instead build a randomized tree/graph through free space, trading optimality for speed — a choice Nav2's global planner and MoveIt2's OMPL backend both have to make.

Appears in: Autonomous Navigation: Theory to Algorithm
Control

Q-Learning (Reinforcement Learning)

Q-learning learns, through trial and error, an estimate Q(state, action) of the long-term reward of taking a given action in a given state, updating that estimate after every experienced transition without needing a model of the environment's dynamics. A persistent Q-table that penalizes previously-bad navigation zones lets a robot carry hard-won experience across separate runs instead of relearning it from scratch each time.

Appears in: MechaBot
Navigation

Occupancy Grids & Costmaps

An occupancy grid discretizes the environment into cells, each holding the probability that it's occupied, built up from LiDAR returns during mapping. Nav2 inflates this into a costmap by adding a decaying cost gradient around obstacles, so the planner is discouraged from grazing walls even where the raw grid is technically free — the same occupancy data also drives coverage planning via distance-transform-based lawnmower paths.

Appears in: MechaBot
Embedded

BLE RSSI Trilateration

Bluetooth Low Energy beacons broadcast a signal whose Received Signal Strength Indicator (RSSI) falls off roughly with the log of distance from the receiver. By measuring RSSI to three or more fixed anchor nodes at known positions and combining the resulting distance estimates, a receiver's 2D/3D position can be trilaterated — noisy in practice, so it's usually paired with filtering (moving average, Kalman) to be usable for asset tracking.

Appears in: BLE Indoor Localisation System

CNNs & Transfer Learning

Convolutional neural networks learn hierarchical spatial filters (edges → textures → parts → objects) directly from image data. Transfer learning starts from a CNN (e.g. ResNet, VGG) already pretrained on a large generic dataset like ImageNet, then fine-tunes it on a much smaller domain-specific dataset (e.g. chest X-rays) — the early, general-purpose filters transfer almost for free, so only the later layers need meaningful retraining.

EKF vs UKF State Estimation

Both the Extended and Unscented Kalman Filters fuse noisy, asynchronous sensor measurements (wheel odometry, IMU, GPS) into a single smoothed state estimate for a nonlinear system. The EKF linearizes the nonlinear model around the current estimate with a first-order Taylor expansion, which breaks down under sharp nonlinearity; the UKF instead propagates a small set of deterministically chosen "sigma points" through the true nonlinear model, typically giving a more accurate estimate at modest extra computational cost.

Appears in: Autonomous Navigation: Theory to Algorithm
Embedded

INT8 Quantization & TFLite Micro

Quantization converts a neural network's 32-bit floating point weights and activations into 8-bit integers, shrinking model size roughly 4× and letting inference run on microcontrollers with no floating-point unit, at dramatically lower latency and power — at the cost of some accuracy, managed by calibrating the int8 ranges against representative data. TensorFlow Lite Micro runs these quantized models in a fixed, pre-allocated "tensor arena" with no dynamic memory allocation, which is what makes it fit on something as constrained as an ESP32.

Appears in: YOLOv8 Edge Inference on ESP32
Embedded

MQTT Telemetry

MQTT is a lightweight publish/subscribe messaging protocol built for constrained devices and unreliable networks: clients publish small messages to named topics through a central broker, and any number of subscribers receive them without the publisher needing to know who's listening. Its low overhead and QoS-level delivery guarantees make it the standard choice for streaming sensor telemetry from IoT devices up to a cloud backend.

Appears in: BLE Indoor Localisation System
