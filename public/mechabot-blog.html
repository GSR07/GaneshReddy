MechaBot: One Nav2 Stack from Gazebo to Real Hardware

How a custom ros2_control hardware interface let the exact same SLAM, localization, and navigation stack drive both a simulated robot and a real one built from an Arduino, two motors, and an RPLIDAR A1.

1. One Nav Stack, Two Worlds

MechaBot is a custom differential-drive autonomous mobile robot built end-to-end: mechanical model, simulation, mapping, localization, navigation, and — the part most projects never reach — the same ROS 2 stack driving real hardware. The goal was simple to state and hard to do: a robot that maps a room, localizes itself, and drives to any goal you click, whether the room is a Gazebo world or my actual floor.

The workspace is organised as seven focused packages:

2. Differential-Drive Kinematics

Everything the robot does rests on one small model. With wheel radius \(r\), wheel separation \(L\), and wheel angular velocities \(\omega_R, \omega_L\):

\[ v = \frac{r\,(\omega_R + \omega_L)}{2}, \qquad \omega = \frac{r\,(\omega_R - \omega_L)}{L} \]

Odometry integrates this forward. Encoder ticks give each wheel's rotation; the pose update per cycle is:

\[ x_{k+1} = x_k + v\,\Delta t \cos\theta_k, \qquad y_{k+1} = y_k + v\,\Delta t \sin\theta_k, \qquad \theta_{k+1} = \theta_k + \omega\,\Delta t \]

In simulation Gazebo produces perfect encoder data. On the real robot this comes from quadrature encoders counted by the Arduino — and every downstream consumer (SLAM, AMCL, Nav2) is identical.

3. The Sim-to-Real Bridge: a Custom ros2_control Hardware Interface

The single most valuable design decision: the robot's motors are hidden behind a custom ros2_control SystemInterface. Nav2 publishes cmd_vel; diff_drive_controller converts it to per-wheel velocity commands; and only at the very bottom does the code fork:

  • Simulation: gz_ros2_control applies the commands inside Gazebo.
  • Hardware: my MechabotInterface streams commands to an Arduino over serial and reads encoder counts back.
# hardware interface, 50 Hz loop (simplified)
def read(time, period):
    counts = serial.request_encoders()          # "e\n" -> "1024,996"
    for i, joint in enumerate(wheels):
        pos[i] = counts[i] * RAD_PER_TICK       # state interface: position
        vel[i] = (pos[i] - prev[i]) / period    # state interface: velocity

def write(time, period):
    l, r = cmd[LEFT], cmd[RIGHT]                # command interface: velocity
    serial.send(f"m {l:.3f} {r:.3f}")           # rad/s per wheel

Swapping sim for real is one launch argument. Nothing above the hardware interface changes — the URDF, the controllers, SLAM, and Nav2 configs are all shared.

4. Mapping and Localization

SLAM Toolbox builds the occupancy grid online from the RPLIDAR A1 scan and wheel odometry. Once a map is saved, AMCL takes over: a particle filter that matches each incoming scan against the map and converges the pose cloud as the robot moves. A global relocalization launch scatters particles across the whole map for the "kidnapped robot" case.

5. Navigation with Recovery Behaviors

Nav2 runs a behavior-tree navigator: global path from the planner server, local trajectory tracking from the controller server, and — critically for a real robot on a real floor — recovery branches (spin, back up, wait, replan) when the robot gets pinched or the costmap disagrees with reality. On top of that sit the autonomy scripts: waypoint patrol, auto-docking with battery monitoring, an aruco-marker detector, and an experimental RL-based docking policy.

Manual override is always live: a joystick feeds twist_mux, which arbitrates between teleop and Nav2 by priority — grab the controller and the autonomy yields instantly.

6. What I'd Tell Past Me

  • Get the hardware interface right first. Every hour spent making the ros2_control layer clean paid back tenfold — the whole nav stack was debugged once, in sim, and simply worked on hardware.
  • Trust the encoders, calibrate the geometry. Odometry drift is dominated by wheel radius and separation errors, not sensor noise. Measure, then measure again.
  • Recovery behaviors are not optional. In sim the robot never gets stuck. On a floor with castors and carpet edges, it does — the behavior tree is what turns "demo" into "leave it running".
