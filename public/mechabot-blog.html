MechaBot: One Nav2 Stack from Gazebo to Real Hardware

The full mobile-robotics pipeline on a custom differential-drive robot: URDF/Xacro modeling, ros2_control with a serial hardware interface, SLAM Toolbox, AMCL, Nav2, camera-based QR docking, battery-aware coverage cleaning, and a Q-learning memory that learns which parts of the map to avoid.

KEY CONCEPTS: Nav2 SLAM AMCL ros2_control Q-Learning Behavior Trees Occupancy Grids

1. What MechaBot Is

MechaBot is a compact differential-drive autonomous mobile robot built end-to-end: robot modeling, Gazebo simulation, real hardware control, SLAM mapping, AMCL localization, Nav2 navigation, QR-code docking, battery-aware mission logic, and an experimental reinforcement-learning layer for adaptive cleaning. The core idea: one software stack that moves between simulation and real hardware with the same ROS 2 interfaces — the robot that maps a Gazebo warehouse is the same robot, bit for bit above the hardware interface, that maps my actual floor.

At a high level the robot can simulate itself in Gazebo, publish TF/odometry/LiDAR/IMU/camera data, build occupancy-grid maps with SLAM Toolbox, localize on saved maps with AMCL, navigate with Nav2 (global planning, local control, costmaps, recoveries), detect QR markers with OpenCV for visual docking, return to a charging dock when battery is low, generate cleaning coverage waypoints directly from the occupancy grid, and learn problematic navigation zones with a persistent Q-table.

2. System Architecture

The stack follows a layered architecture — sensors and actuators at the bottom, task autonomy at the top. Every layer talks to its neighbors only through standard ROS 2 interfaces, which is what makes the sim/real swap possible.

The frame chain

Everything in the stack hangs off one TF chain:

\[ \text{map} \;\to\; \text{odom} \;\to\; \text{base\_footprint} \;\to\; \text{base\_link} \;\to\; \text{sensors/wheels} \]

map → odom comes from localization (SLAM Toolbox while mapping, AMCL afterwards), odom → base_footprint comes from wheel odometry via diff_drive_controller, and the fixed robot geometry comes from the URDF through robot_state_publisher. When people say "my robot jumps around in RViz", they are almost always describing a disagreement between the first two transforms.

3. The Robot Model

The robot is defined in mechabot.urdf.xacro, split into base, sensors, Gazebo properties, and ros2_control modules. The key design decision is a single argument that switches the entire control backend:

<xacro:arg name="is_sim" default="true"/>

With is_sim:=true the model loads the Gazebo/Ignition ros2_control plugin; with is_sim:=false it loads my custom serial hardware plugin from mechabot_firmware. Nothing else in the file changes.

The physical model has two driven wheels, four castors, a top lid, and three sensor frames (lidar_link, camera_link + optical frame, imu_link), with realistic masses (5 kg base, 2 kg lid, 0.5 kg per wheel) so Gazebo's physics behave sensibly. The geometry that matters for everything downstream:

ParameterValue
Wheel radius \(r\)0.034 m
Wheel separation \(L\)0.185 m (wheel joints at \(y = \pm 0.0925\) m)
Footprint0.270 m × 0.230 m rectangle
Wheel jointscontinuous, axis \((0, 1, 0)\)

One subtlety worth calling out: the camera optical frame uses the standard optical convention rotation rpy="-π/2 0 -π/2", so OpenCV sees +z forward / +x right / +y down, regardless of how the physical camera link is oriented on the robot.

4. Differential-Drive Kinematics

Everything the robot does rests on one small model. With wheel radius \(r\), separation \(L\), and wheel angular velocities \(\omega_L, \omega_R\), the wheel rim speeds are \(v_{L,R} = r\,\omega_{L,R}\), and the body velocities follow:

\[ v = \frac{r\,(\omega_R + \omega_L)}{2}, \qquad \omega = \frac{r\,(\omega_R - \omega_L)}{L} \]

The controller needs the inverse map — body command in, wheel commands out:

\[ \omega_R = \frac{v + \omega L/2}{r}, \qquad \omega_L = \frac{v - \omega L/2}{r} \]

Plugging in MechaBot's numbers makes it concrete. Driving straight at \(v = 0.15\) m/s: both wheels spin at \(0.15/0.034 = 4.41\) rad/s. Rotating in place at \(\omega = 1.0\) rad/s: the wheels split to \(\pm(0.185/2)/0.034 = \pm 2.72\) rad/s. This same algebra is behind diff_drive_controller, wheel odometry, in-place rotations, and the QR alignment controller later.

Odometry integration

Wheel feedback is integrated into a planar pose \((x, y, \theta)\) at every control cycle:

\[ x_{t+1} = x_t + v\cos\theta_t\,\Delta t, \qquad y_{t+1} = y_t + v\sin\theta_t\,\Delta t, \qquad \theta_{t+1} = \theta_t + \omega\,\Delta t \]

This is the entire odom → base_footprint transform. It drifts — every small error in \(r\) or \(L\) integrates forever — which is exactly why the map frame and a localizer exist.

The real hardware interface

The sim-to-real bridge is a C++ hardware_interface::SystemInterface plugin in mechabot_firmware. It talks to the embedded motor controller over serial at 500000 baud on /dev/ttyUSB1, exporting velocity command interfaces and position + velocity state interfaces for both wheels. Commands go out as compact formatted strings:

rp05.30,ln12.45,

meaning right wheel, positive, 5.30 rad/s; left wheel, negative, 12.45 rad/s — the pattern is r<sign><velocity>,l<sign><velocity>, with two decimals. Feedback comes back in the same style and is parsed into signed wheel velocities. Since the embedded side reports velocity, joint position is recovered by integrating in the plugin's read():

// theta_i(t + dt) = theta_i(t) + omega_i(t) * dt
position_states_[i] += velocity_states_[i] * dt;

The payoff: the ROS side is completely independent of the embedded implementation. Swapping sim for real is one launch argument, and the URDF, controllers, SLAM, AMCL, and Nav2 configs are all shared. The whole nav stack was debugged once, in sim, and simply worked on hardware.

6. Sensors: Simulated Like the Real Ones

Gazebo sensors are configured to match the real hardware's character, not an idealised version of it:

SensorConfiguration
LiDAR (/scan)GPU LiDAR, 360 samples over \(2\pi\), 5 Hz, range 0.15–12.0 m, 0.02 m resolution, Gaussian noise \(\sigma = 0.01\) — mirroring the RPLIDAR A1
IMU (/imu/out)100 Hz; quaternion converted to yaw for rotation control
Camera (/camera/image_raw)640 × 480 @ 10 Hz, 1.396 rad horizontal FOV, clip 0.02–8.0 m

Each LiDAR beam is a polar measurement \(z_i = (r_i, \phi_i)\) with \(\Delta\phi \approx 2\pi/360\). The autonomy nodes derive a single scalar from it — the front obstacle distance, taken as the minimum (or mean, in simpler nodes) of the valid ranges in a small sector around 0° — used for emergency stops and docking thresholds. For yaw control, the IMU quaternion collapses to:

\[ \text{yaw} = \operatorname{atan2}\!\big(2(wz + xy),\; 1 - 2(y^2 + z^2)\big) \]

7. Mapping with SLAM Toolbox

Mapping uses SLAM Toolbox with the Ceres solver (SPARSE_NORMAL_CHOLESKY + Levenberg–Marquardt), scan matching and loop closure enabled, max laser range 12 m, and a map resolution of 0.05 m/cell. Conceptually, graph SLAM estimates the set of robot poses \(X\) that best explains all constraints simultaneously:

\[ X^\ast = \arg\min_X \sum_i \left\lVert e_i(X) \right\rVert^2_{\Omega_i} \]

where each \(e_i\) is the error of an odometry, scan-matching, or loop-closure constraint and \(\Omega_i\) weights how much you trust it. Loop closures are what pull the accumulated odometry drift back into shape — without them, the two ends of a long corridor never quite meet.

Occupancy grid math

The output map stores each cell as unknown (−1), free (0), or occupied (100). Converting a grid cell \((m_x, m_y)\) back to world coordinates — needed later when the cleaning planner turns cells into waypoints — is a scale plus the map-origin rigid transform:

\[ x_{local} = (m_x + 0.5)\,\text{res}, \qquad y_{local} = (m_y + 0.5)\,\text{res} \] \[ x_{world} = x_0 + x_{local}\cos\theta_0 - y_{local}\sin\theta_0, \qquad y_{world} = y_0 + x_{local}\sin\theta_0 + y_{local}\cos\theta_0 \]

The project ships saved maps for the small_house and qr_maze Gazebo worlds, both at 0.05 m/cell.

8. Localization with AMCL

Once a map exists, AMCL estimates the posterior over the robot pose with a particle filter:

\[ p(x_t \mid z_{1:t}, u_{1:t}, m) \]

where \(x_t\) is the pose, \(z_t\) the LiDAR scan, \(u_t\) the odometry motion, and \(m\) the known map. Each cycle repeats three steps over the particle set:

\[ \textbf{1. Motion:}\quad x_t^{[i]} \sim p(x_t \mid u_t, x_{t-1}^{[i]}) \qquad \textbf{2. Weight:}\quad w_t^{[i]} = p(z_t \mid x_t^{[i]}, m) \qquad \textbf{3. Resample by } w_t \]

MechaBot uses the DifferentialMotionModel with 500–2000 particles, the likelihood_field laser model with 60 beams, and update thresholds of 0.25 m / 0.2 rad so the filter only spends compute when the robot has actually moved. The likelihood-field model scores a particle by how close each projected beam endpoint lands to an obstacle in the map — a simplified per-beam probability:

\[ p(z \mid x, m) = z_{hit}\, e^{-d^2 / (2\sigma_{hit}^2)} + \frac{z_{rand}}{z_{max}}, \qquad \sigma_{hit} = 0.2,\; z_{hit} = z_{rand} = 0.5 \]

where \(d\) is the distance from the measured endpoint to the nearest obstacle. The initial pose is seeded at the dock (\(x = 1.5,\; y = 5.18,\; \theta = 1.57\)), so the robot wakes up knowing roughly where it is.

9. Navigation: Nav2 End to End

Nav2 brings up planner_server, controller_server, smoother_server, behavior_server, bt_navigator, and waypoint_follower under a lifecycle manager.

Global planning — SmacPlanner2D

The global planner is grid-based A*-style search over the costmap, minimizing the classic:

\[ f(n) = g(n) + h(n) \]

with \(g\) the accumulated path cost and \(h\) the heuristic to goal. A cost-travel multiplier of 2.0 makes cells near inflated obstacles genuinely expensive, so the planner prefers open space over wall-scraping shortcuts. Tolerance is 0.125 m with a 2 s planning budget.

Local control — Regulated Pure Pursuit

The local controller picks a lookahead point on the global path and steers toward it. If that point is at \((x_L, y_L)\) in robot coordinates at lookahead distance \(L_d\), the commanded curvature and angular velocity are:

\[ \kappa = \frac{2\,y_L}{L_d^2}, \qquad \omega = v\,\kappa \]

The "regulated" part scales speed down in tight curvature and near obstacles, and adds collision checking. MechaBot runs it deliberately slow and precise: 0.15 m/s desired velocity, adaptive lookahead 0.30–0.50 m (1.5 s lookahead time), rotate-to-heading at 1.8 rad/s, reversing allowed, and tight goal tolerances of 0.04 m / 0.04 rad — because the docking approach pose needs to be exact.

Costmaps and inflation

Both costmaps run at 0.05 m/cell with the rectangular footprint. The inflation layer (radius 0.30 m, cost scaling 4.0) decays cost exponentially with distance from obstacles:

\[ \text{cost}(d) = C\, e^{-k\,(d - r_{inscribed})} \]

This buffer is what turns "collision-free" paths into comfortable paths. On top sits the behavior tree (simple_navigation_w_replanning_and_recovery.xml) with Spin, BackUp, and Wait recoveries — in sim the robot never gets stuck; on a floor with castors and carpet edges it does, and recoveries are what turn "demo" into "leave it running". Manual override stays live throughout: a joystick feeds twist_mux, which arbitrates teleop vs. Nav2 by priority.

10. QR-Code Docking: Visual Servoing for the Last 60 cm

Nav2 is good at reaching a pose, but charging contacts need alignment tighter than a 4 cm map-frame tolerance can guarantee. So docking is split into two phases: Nav2 gets the robot to a pre-dock pose, then a camera + LiDAR controller takes over for the final approach.

stateDiagram-v2 [*] --> Undock: reverse fixed distance Undock --> Mission: rotate away, patrol / clean Mission --> PreDock: battery low or route done
(Nav2 to pre-dock pose) PreDock --> Acquire: search for QR marker Acquire --> Align: marker visible Align --> Approach: pixel error small Approach --> Docked: LiDAR d ≤ 0.15 m,
max travel, or timeout Docked --> [*]: charging

The pre-dock pose

The pre-dock pose is computed by backing away from the dock pose \((x_{dock}, y_{dock}, \theta_{dock})\) along the dock heading:

\[ x_{pre} = x_{dock} - d\cos\theta_{dock}, \qquad y_{pre} = y_{dock} - d\sin\theta_{dock} \]

With the dock at \((1.5,\, 5.18,\, 1.57)\) and \(d = 0.60\) m, this lands at roughly \((1.5,\, 4.58)\) — 60 cm straight out from the charger, facing it.

Proportional visual servoing

For docking the robot never decodes the QR payload — cv2.QRCodeDetector.detect() returns only the four corners, and the controller needs just their centroid \(c_x\). The horizontal pixel error against the image center drives yaw:

\[ e = c_x - \frac{\text{image width}}{2}, \qquad \omega_z = \operatorname{clip}(-K_p\, e,\; \omega_{min},\; \omega_{max}) \]

Different phases use different gains: \(K_p = 0.0030\) for coarse alignment, 0.0015 in the safer battery-docking version, and 0.0010 during the final forward creep — the clip prevents oscillation when the marker sits far off-center. Any rotation to a target yaw uses the same wrapped-error proportional law you'd use on any robot:

\[ e_\theta = \operatorname{atan2}\!\big(\sin(\theta_{target} - \theta),\, \cos(\theta_{target} - \theta)\big), \qquad \omega_z = \operatorname{clip}(K_p\, e_\theta,\, -0.35,\, 0.35) \]

The atan2 wrap keeps \(e_\theta \in [-\pi, \pi]\) so the robot never rotates the long way around; it stops when \(|e_\theta| < 0.035\) rad (about 2°).

Layered stop conditions

The final approach never trusts one sensor. The robot creeps forward while correcting heading and stops on any of: LiDAR front distance \(\le 0.15\) m, odometry travel exceeding a max distance from the approach start, a timeout, or (in some versions) the QR marker growing past a size threshold. One more real-world fix lives here: the QR marker drops out for a few frames under motion blur, so the controller keeps a 0.7 s QR memory — briefly losing the marker continues on the last known center instead of aborting.

11. The QR Maze Solver

The same camera reads QR payloads (left, right, stop) posted around a Gazebo maze, this time via detectAndDecode(). The node is a small state machine — FORWARD → TURNING → FORWARD, or FORWARD → STOPPED. It drives at 0.15 m/s until the LiDAR front distance drops below 0.45 m, then executes the last QR command it saw: a left turn targets \(\theta + \pi/2\), a right turn \(\theta - \pi/2\) (from IMU yaw, using the wrapped P-rotation above), and stop halts permanently. It's the same three primitives — drive, rotate-to-yaw, read the world — recombined.

12. Battery-Aware Cleaning and Coverage Planning

The most complete mission is a vacuum-style cleaning system. A simulated battery drains at 0.25 %/s while working and charges at 2.0 %/s on the dock:

\[ B_{t+1} = B_t - d_{drain}\,\Delta t \;\;\text{(working)}, \qquad B_{t+1} = B_t + d_{charge}\,\Delta t \;\;\text{(charging)} \]

When \(B\) drops below 45%, the node cancels the current Nav2 goal, returns to the pre-dock pose, runs the full QR/LiDAR docking sequence, charges to 100%, undocks, and resumes the same cleaning route where it left off.

Coverage waypoints straight from the map

Cleaning waypoints aren't hand-placed — they're generated from the live /map occupancy grid in a pipeline:

  1. Threshold free space: a cell is free if \(0 \le occ \le 25\); unknown cells count as obstacles.
  2. Run an OpenCV distance transform: \(D(m_x, m_y)\) = distance to the nearest obstacle cell, converted to meters via the 0.05 m resolution.
  3. Keep only cells with clearance \(D_m \ge 0.40\) m — waypoints never hug walls or clutter.
  4. Sweep rows in a lawnmower (boustrophedon) pattern: with 0.65 m coverage spacing, the row step is \(\lfloor 0.65 / 0.05 \rfloor = 13\) cells, alternating direction each row to minimise transit.
  5. Convert surviving cells to world coordinates (the grid-to-world transform from §7).
  6. Drop waypoints within 1.20 m of the dock — \(\sqrt{(x - x_{dock})^2 + (y - y_{dock})^2} < 1.20\) is forbidden — and de-duplicate points that are too close together.

The result is a safe, evenly spaced coverage route computed in milliseconds from whatever map the robot happens to have.

13. Learning the Bad Zones: A Q-Table Over Nav2

Some map regions fail repeatedly — clutter, narrow passages, costmap artifacts. Rather than replacing Nav2, the RL layer sits above it and learns how to dispatch waypoints, persisting everything to a JSON memory (cleaning_rl_memory.json) so learning survives restarts.

State, actions, policy

The state key discretises the waypoint to a 0.50 m grid — \(x_g = \operatorname{round}(x/s)\,s\) — and appends the front-obstacle band, whether the point is near a learned bad zone, and the battery band, e.g. x=-4.5|y=2.0|front=clear|bad=0|battery=ok. Three actions are available per waypoint: NAVIGATE (send to Nav2 directly), CLEAR_AND_NAVIGATE (clear costmaps first), and SKIP. Selection is ε-greedy with \(\varepsilon = 0.08\) — mostly exploit the best known action, occasionally explore.

The update

\[ Q(s, a) \leftarrow Q(s, a) + \alpha\left[ r + \gamma \max_{a'} Q(s', a') - Q(s, a) \right], \qquad \alpha = 0.35,\; \gamma = 0.80 \]
OutcomeReward
SUCCEEDED+10
LOW_BATTERY (interrupted)0
SKIPPED−2
FAILED−30
TIMEOUT−35
COLLISION_RISK−80

On failure, timeout, or collision risk the node also records a circular bad zone of radius 0.75 m around the waypoint; future waypoints falling inside a learned bad zone can be skipped automatically. Over a few runs the robot stops re-attempting the corner it always gets pinched in — cheap, persistent, and it never fights the planner.

14. Challenges That Shaped the Design

  • Stable docking near the charger. Global navigation isn't precise enough for physical charging contacts. Solution: split docking into Nav2-to-pre-dock plus QR/LiDAR visual servoing for the final centimeters.
  • QR detection dropouts. Motion blur and viewing angle kill detection for a few frames. Solution: the 0.7 s QR memory — hold the last known marker center instead of aborting or oscillating.
  • Unsafe coverage waypoints. Naïve grid sampling puts waypoints against walls. Solution: distance-transform clearance filtering (\(D_m \ge 0.40\) m) before waypoint generation.
  • Repeated navigation failures. Some zones fail every run. Solution: the Q-table plus persistent bad-zone memory changes the dispatch decision, not the planner.
  • Sim-to-real drift. Odometry error is dominated by wheel radius and separation calibration, not sensor noise. Measure the geometry, then measure it again — and keep both backends behind the same ros2_control interfaces so nothing above them ever knows the difference.

15. Where It Goes Next

The natural upgrades: fuse wheel odometry and IMU with an EKF, replace the simulated battery with real voltage sensing, move from QR to AprilTags with full pose-from-corners visual servoing for docking, track cleaned-area coverage explicitly, and train the RL policy properly in simulation before deploying it. But the architecture is already the point — every one of those swaps happens behind an interface the rest of the stack never sees.
