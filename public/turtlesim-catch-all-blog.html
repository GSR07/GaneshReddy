Turtlesim Catch-Them-All: Every Core ROS 2 Concept in ~200 Lines

A spawner node, a hunter node, three custom interfaces, and a 100 Hz proportional controller — how one small turtlesim project exercises the entire ROS 2 communication model: topics, services, async clients, timers, and custom message design.

KEY CONCEPTS: ROS 2 Nodes/Topics/Services P-Control

1. Why Turtlesim Is the Right Gym

Every ROS 2 system — a warehouse AMR, a 7-DOF manipulator, a fleet manager — is built from the same five primitives: nodes, topics, services, interfaces, and timers. The problem with learning them on a real robot is that hardware noise, TF trees, and launch-file plumbing bury the concepts. Turtlesim strips all of that away: a 11×11 unit 2D world, a turtle with a perfect pose sensor, and a cmd_vel interface identical in shape to what drives a real differential-drive robot.

The game: turtles spawn at random poses every two seconds; a hunter turtle must chase each one down and "catch" it (remove it from the world). Simple to state — but implementing it properly forces you through every communication pattern ROS 2 offers, including the ones tutorials usually skip: custom interface packages, asynchronous service clients, and closed-loop control inside a timer callback.

2. Architecture: Two Nodes and a Referee

The system is three nodes. turtlesim_node is the "world" — it owns the turtles and exposes standard services. My two nodes divide the game logic cleanly:

NodeOwnsCommunication
turtle_spawnerThe list of alive turtles (single source of truth)Publishes alive_turtles, serves catch_turtle, calls /spawn and /kill
turtle_controllerThe hunter's control loopSubscribes to /turtle1/pose and alive_turtles, publishes cmd_vel, calls catch_turtle

Note the deliberate asymmetry: the controller never talks to /kill directly. It requests a catch from the spawner, and the spawner — the owner of the alive list — decides, kills, and republishes. That is the ROS pattern for shared state: one node owns the data; everyone else reads it from a topic or mutates it through a service. Two nodes both calling /kill and both keeping their own turtle lists would race and drift apart.

3. Topics vs Services: Choosing the Right Primitive

The project uses both communication styles, and the choice is never arbitrary:

  • Topics (pub/sub, many-to-many, fire-and-forget) carry streams: the turtle pose at ~60 Hz, velocity commands at 100 Hz, and the alive-turtle list whenever it changes. Nobody waits for an answer.
  • Services (request/response, one-to-one) carry transactions: "spawn a turtle named turtle7 at (3.2, 8.1)" has a definite outcome the caller needs — success or failure, plus the assigned name.

A useful litmus test that generalises far beyond turtlesim: if losing one message doesn't matter, it's a topic; if you need to know it happened, it's a service. A missed pose update is irrelevant — a fresher one arrives in 16 ms. A missed kill request leaves a ghost turtle in the game forever.

4. Custom Interfaces: rob_interfaces

Standard messages don't describe "a named turtle at a pose", so the project defines its own interface package — an ament_cmake package containing nothing but message and service definitions:

# msg/Turtle.msg
string name
float64 x
float64 y
float64 theta

# msg/TurtleArray.msg
Turtle[] turtles          # messages compose: an array of a custom type

# srv/CatchTurtleKill.srv
string name               # ── request
---
bool success              # ── response

Three design points worth noting:

  • Interfaces live in their own package. Both Python nodes depend on rob_interfaces, but the interface package depends on neither. If tomorrow a C++ node or a dashboard wants the turtle list, it links against the same package — interfaces are the API contract of the system.
  • Message composition: TurtleArray embeds Turtle[], exactly how PoseArray embeds Pose in the standard library. The code generator handles nesting for free.
  • Service responses should carry outcomes. CatchTurtleKill returns a bool success, which the controller checks and logs — a failed catch is an error, not a silent no-op.

5. The Spawner: Timers, Async Clients, and Why You Never Block

The spawner's rhythm comes from a wall timer: every 2 seconds, generate a random pose and call /spawn:

self.spawn_turtle_timer_ = self.create_timer(2.0, self.spawn_new_turtle)

def spawn_new_turtle(self):
    self.turtle_counter_ += 1
    name  = "turtle" + str(self.turtle_counter_)
    x     = random.uniform(0.0, 11.0)      # turtlesim world is 11 x 11
    y     = random.uniform(0.0, 11.0)
    theta = random.uniform(0.0, 2 * math.pi)
    self.call_spawn_service(name, x, y, theta)

The interesting part is how the service is called. ROS 2 offers call() (synchronous) and call_async() (returns a future). Inside a node callback you must use the async form:

future = self.spawn_client_.call_async(request)
future.add_done_callback(
    partial(self.callback_call_spawn_service, request=request))

The reason is a classic deadlock. A single-threaded executor processes one callback at a time. A synchronous call() inside the timer callback would block the executor thread waiting for a response — but that same thread is the only one that can receive the response. The node freezes forever. call_async() returns immediately; the executor stays free; and when the response arrives, the done-callback fires with the result.

functools.partial solves the second subtlety: the response to /spawn doesn't echo back the pose you asked for, so the done-callback binds the original request. Only when the world confirms the spawn does the turtle enter the alive list and get republished:

def callback_call_spawn_service(self, future, request):
    response = future.result()
    if response.name != "":
        new_turtle       = Turtle()
        new_turtle.name  = response.name
        new_turtle.x     = request.x        # bound via partial — the
        new_turtle.y     = request.y        # response doesn't carry the pose
        new_turtle.theta = request.theta
        self.alive_turtles_.append(new_turtle)
        self.publish_alive_turtles()

6. The Controller: a 100 Hz Proportional Controller

The hunter runs a control loop on a 10 ms timer. Each tick computes the error between the hunter's pose \((x, y, \theta)\) and the target \((x_t, y_t)\), and publishes a velocity command. Turtlesim's turtle is a unicycle model — it can only move along its heading and rotate:

\[ \dot{x} = v\cos\theta, \qquad \dot{y} = v\sin\theta, \qquad \dot{\theta} = \omega \]

So the controller has exactly two knobs, \(v\) (linear) and \(\omega\) (angular), and two errors to drive to zero — distance and heading:

\[ d = \sqrt{(x_t - x)^2 + (y_t - y)^2}, \qquad \theta_{goal} = \operatorname{atan2}(y_t - y,\; x_t - x) \] \[ v = K_v\, d, \qquad \omega = K_\omega \,\big(\theta_{goal} - \theta\big)_{wrapped}, \qquad K_v = 2,\; K_\omega = 6 \]

This is pure proportional control: command strength scales with error. Far from the target the turtle sprints; close in, it slows down smoothly, so it doesn't need a separate braking phase. Two details make it work:

atan2, not atan

\(\operatorname{atan2}(\Delta y, \Delta x)\) returns the bearing in the correct quadrant over the full \((-\pi, \pi]\) range. A plain \(\arctan(\Delta y / \Delta x)\) collapses opposite directions onto the same angle and divides by zero when the target is straight above — the classic bug that sends robots driving away from their goal.

Wrapping the heading error

Angles live on a circle. If the hunter faces \(\theta = 3\) rad and the goal bearing is \(-3\) rad, the raw difference is \(-6\) rad — commanding nearly a full spin the long way round, when the true error is only \(\approx 0.28\) rad the other way. The fix normalises the error into \([-\pi, \pi]\):

diff = goal_theta - self.pose_.theta
if diff > math.pi:
    diff -= 2 * math.pi          # take the short way round
elif diff < -math.pi:
    diff += 2 * math.pi
cmd.angular.z = 6 * diff

Skip this and the turtle visibly pirouettes before every catch. With it, the hunter always turns through the minimal angle — and because \(K_\omega \gg K_v\), heading converges faster than position, so the trajectory bends into a clean pursuit curve rather than a spiral.

The catch condition

When \(d \le 0.5\) the controller declares the turtle caught: it zeroes the command, fires the catch_turtle service (async, of course — this runs inside the timer callback), and clears its target until the next alive_turtles update assigns a new one:

if distance > 0.5:
    cmd.linear.x  = 2 * distance
    cmd.angular.z = 6 * diff
else:
    cmd.linear.x  = 0.0
    cmd.angular.z = 0.0
    self.call_catch_turtle_service(self.turtle_to_catch_.name)
    self.turtle_to_catch_ = None

7. The Catch Handshake, End to End

The full lifecycle of one catch shows every primitive cooperating:

sequenceDiagram participant C as turtle_controller participant S as turtle_spawner participant W as turtlesim_node C->>C: control loop reaches d ≤ 0.5 C->>S: catch_turtle(name) — async request S->>W: /kill(name) — async request W-->>S: kill confirmed S->>S: remove from alive list S-->>C: alive_turtles (topic) — updated list C->>C: lock onto next target

Nothing blocks anywhere in this chain. The controller keeps its 100 Hz loop alive while the catch request is in flight; the spawner keeps spawning while the kill request is in flight. That responsiveness under concurrency is not a nicety — it is exactly the property that lets the same patterns scale to a robot where the "control loop" is Nav2 and the "service" is a docking request.

8. What This Small Project Actually Teaches

  • Ownership beats sharing. One node owns the mutable state; topics broadcast it; services mutate it. This single rule eliminates whole classes of race conditions.
  • Async service calls are not optional. The sync-call-in-callback deadlock is the most common beginner ROS 2 freeze, and this project makes the correct future/done-callback pattern muscle memory.
  • Control theory shows up immediately. Even a toy pursuit needs a proportional law, angle wrapping, and gain tuning — the same P-on-heading structure I later reused for real differential-drive robots in the MechaBot and TurtleBot3 projects.
  • Interface packages are the system's contract. Defining Turtle, TurtleArray, and CatchTurtleKill in a standalone package is the habit that keeps larger workspaces language-agnostic and dependency-clean.
