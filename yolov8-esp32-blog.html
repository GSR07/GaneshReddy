YOLOv8 Edge Inference on ESP32

Challenges and breakthroughs from porting lighter convolutional architectures to bare-metal embedded devices with tight RAM constraints.

1. The Challenge of Bare-Metal Edge AI

Running a state-of-the-art object detection model like YOLOv8 on a standard PC is trivial. Running it on an ESP32 microcontroller with a few megabytes of PSRAM and barely any flash storage? That's an engineering battle. The constraint isn't just compute speed, it's extreme memory limitation.

In this post, I explore how to strip down, quantize, and optimize YOLO architectures into TensorFlow Lite Micro (.tflite) payloads suitable for bare-metal C++ execution.

2. Model Reduction & INT8 Quantization

The standard YOLOv8n (Nano) is still too large. We have to prune the network, reduce input resolutions to 96x96 or 160x160, and perform Post-Training Quantization (PTQ) to convert all 32-bit floating-point weights into 8-bit integers.

\[ r = S (q - Z) \]

Quantization Math: We map the real value \(r\) to the quantized integer \(q\) using a Scale \(S\) and a Zero-point \(Z\). This drastically reduces the memory footprint by 4x and speeds up inference on integer-only ALUs.

3. TensorFlow Lite Micro Implementation

Once converted to a C-style array using xxd -i model.tflite > model_data.cc, we use the TensorFlow Lite Micro library in ESP-IDF to allocate a tensor arena and invoke the interpreter.

// 1. Define memory arena
constexpr int kTensorArenaSize = 1024 * 500; // 500 KB limit!
uint8_t tensor_arena[kTensorArenaSize];

// 2. Setup the MicroInterpreter
tflite::MicroInterpreter interpreter(
    model, resolver, tensor_arena, kTensorArenaSize, error_reporter);

// 3. Allocate and map image data to input tensor
TfLiteTensor* input = interpreter.input(0);
memcpy(input->data.int8, camera_fb->buf, camera_fb->len);

// 4. Run Inference!
if (interpreter.Invoke() != kTfLiteOk) {
    ESP_LOGE("TF", "Invoke failed");
}

4. Memory Management & Heap Limitations

The ESP32 typically features 520KB of internal SRAM. A large model will immediately fail on AllocateTensors(). The trick is to utilize external PSRAM (e.g., the 4MB module on ESP32-CAM boards). However, PSRAM is connected via SPI and is vastly slower than internal SRAM.

By carefully pinning the TFLite operations to internal memory and storing only the static weights in PSRAM/Flash via the EXT_RAM_ATTR directive, we strike a working balance between capacity and latency, achieving ~1-2 FPS for basic real-time object detection directly on the sensor node.
